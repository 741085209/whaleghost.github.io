线程构造器

构造器 说明
thread() noexcept; 默认构造函数
thread( thread&& other ) noexcept; 移动构造函数
template<class F, class… Args>
explicit thread( F&& f, Args&&… args );		 带参构造函数，f为可调用对象，args为可变参数列表
thread( const thread& ) = delete; 拷贝构造函数被禁用

构造参数

std::thread在调用template<class F, class... Args> explicit thread( F&& f, Args&&... args )时，args被捕获时，分为以下情况：

  • lvalue：args被拷贝构造
  • rvalue/xvalue：args被移动构造

无论何种情况都是传递副本，若有改变args值的需求，需为引用传递，需用std::ref或std::cref包装器进行包装。

构造细节

std::decay拿到原始类型后根据传入参数的值类型调用拷贝/移动构造器

传入的f与args会经过std::decay退化为原类型，在构造时会执行退化得到原类型并进行拷贝并移入std::thread中。

换言之，子线程捕获了构造时的f与args，并单独持有一份，至于执行何种构造/拷贝，需要看具体类型与拷贝构造函数实现。

std::thread arguments must be invocable after conversion to rvalues

并非所有对象都支持移动语义，因此在将对象作为参数传递给std::thread构造函数时，需要确保这些对象在转换为右值后仍然可以调用。

这就导致当传入子线程的可调用对象的参数包含对象引用时，必须传入可接取右值的引用，T&&或const T&。（传入std::ref包装器时此条不适用）

detach下的常见危险行为

所有危险行为都由生命周期引起，在使用detach时需细致检查生命周期问题。或压根不用detach。

子线程执行detach后将挂到后台独立执行，被称为守护线程，已分配的资源由自身进行管理释放。

可调用对象内部含有引用/指针

本质上是由于浅拷贝引起的悬垂引用，多个对象同时引用了同一块内存（非特指堆内存，也包括栈内存），只要其中一个对象析构，那么其余对象必将产生悬垂引用，从而产生未定义行为（UB）。可通过定义深拷贝构造函数解决。

使用全局对象

std::cout为全局对象，生命周期与主线程相同

参数含有指针

我们需要明确，T*经过std::decay后仍是T*，所以std::thread在构造时捕获的仍然是T*，无法生成其拷贝，而引用T&与T&&在经过std::decay后均可得到T。

我们可以看到&i与参数s的值相同，这种情况下我们在print中使用s极易产生UB。

由于类型转换导致decay无法拿到copy

参数涉及类型转换时，有两种情况：

子线程转换构造（非良构）

问题：

  1. 参数传入了数组，std::decay后是T*，同参数含有指针情况。
  2. 主线程将参数分配给子线程后会继续执行，子线程进行char[]->std::string时可能主线程已执行完毕，资源已释放，又由于char[]经过std::decay后是char*，无法拿到copy，根据已释放掉的资源去转换构造产生UB。

主线程转换构造（良构）

主线程转换构造而不是子线程，这样不必担心资源被释放掉而转换构造尚未完成。

总结

detach的使用限制非常多，且官方对其规范甚少，绝大多数情况下极易产生UB，我们应避免在工程中使用detach，或者直接禁止使用detach。

如果我们不用detach，以上所有的问题都可忽略。

  • 尽量不要出现子线程负责类型转换构造
  • 尽量不要出现向thread中传递指针or裸数组（退化后是指针）

std::lock_guard

源码（GCC）

构造器 说明
template<typename T>
lock_guard(T& m);		 根据锁构造lock_guard。
template<typename T>
lock_guard(const lock_guard&) = delete;		 禁用拷贝构造。

我们会发现std::mutex的使用酷似指针，lock=>new，unlock=>delete，如果忘记unlock/delete会导致极其严重的后果，分别对应死锁与内存泄露。那么我们可以用RAII思想来避免这种问题。std::lock_guard正是运用了这种思想。

RAII思想：

  1. 在对象的构造函数中获取资源。
  2. 在对象的析构函数中释放资源。

这种方法确保了资源的正确获取和释放，避免了因忘记释放资源而导致的内存泄漏或其他问题。

使用lock_guard后不要再对lock_guard所保护的锁进行操作。

按顺序加锁

std::lock

锁定给定的可锁定对象 lock1、lock2、…、lockn (n >= 3)，使用免死锁算法以避免死锁。锁定指定的互斥体，若任何一个不可用则阻塞。

对 lock、try_lock 和 unlock 的未指定系列调用锁定对象。若调用 lock 或 unlock 导致异常，则在重抛前对任何已锁的对象调用 unlock。

使用std::lock后再使用RAII风格锁时应在第二参数传入std::adopt_lock，表示已有当前互斥锁的所有权，否则已占有此 mutex 的线程会再次调用 锁的lock，导致死锁，UB。

std::scoped_lock

std::scoped_lock是一个RAII风格的锁，它可以同时获取多个锁，并在作用域结束时自动释放。
